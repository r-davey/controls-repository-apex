# Oracle SQL vs PostgreSQL — Reference Sheet

Personal reference for the Oracle APEX re-platform of the Controls Repository.
Every difference I've hit while translating Phase 1 of the PostgreSQL project.

---

## 1. Data types

Oracle renames most of the types I used in PostgreSQL. This is the single 
most common source of syntax errors when translating.

| Concept                   | PostgreSQL                    | Oracle                                |
|---------------------------|-------------------------------|---------------------------------------|
| Short variable string     | `VARCHAR(100)`                | `VARCHAR2(100)`                       |
| Long text / unbounded     | `TEXT`                        | `CLOB`                                |
| Integer                   | `INTEGER`, `INT`              | `NUMBER` or `NUMBER(10)`              |
| Decimal / numeric         | `NUMERIC(10,2)`               | `NUMBER(10,2)`                        |
| Auto-increment PK         | `SERIAL`, `BIGSERIAL`         | `NUMBER` + SEQUENCE + TRIGGER (see §2)|
| Boolean                   | `BOOLEAN`                     | `NUMBER(1) CHECK (col IN (0,1))`      |
| Date only                 | `DATE`                        | `DATE` (note: Oracle DATE includes time) |
| Timestamp                 | `TIMESTAMP`                   | `TIMESTAMP`                           |
| Timestamp w/ timezone     | `TIMESTAMP WITH TIME ZONE`    | `TIMESTAMP WITH TIME ZONE`            |
| JSON storage              | `JSONB`                       | `CLOB` (optionally with `IS JSON` check) |

### Key gotchas

- **`VARCHAR` vs `VARCHAR2`** — always use `VARCHAR2` in Oracle. `VARCHAR` is 
  technically supported but deprecated; every real Oracle codebase uses `VARCHAR2`.
- **No native BOOLEAN in table columns** — use `NUMBER(1)` with a `CHECK` 
  constraint. Oracle 23ai introduced a real BOOLEAN type but tooling support 
  is inconsistent; sticking with the `NUMBER(1)` convention for this project.
- **Oracle `DATE` includes time** — it's actually a datetime despite the name. 
  Use `TIMESTAMP` when you need fractional seconds; use `DATE` when day-level 
  precision is enough.
- **JSON lives in `CLOB`** — no dedicated `JSONB` type. You can add an 
  `IS JSON` check constraint to enforce validity, which we'll do in P1-09 
  for `control_versions` and `audit_log`.

## 2. Identity / auto-increment

PostgreSQL's `SERIAL` is a convenience wrapper around a sequence + default 
value. Oracle makes you build the equivalent in three explicit pieces: 
a `NUMBER` column, a `SEQUENCE`, and a `BEFORE INSERT` trigger.

### PostgreSQL (what I used before)

```sql
CREATE TABLE roles (
    role_id   SERIAL PRIMARY KEY,
    role_name VARCHAR(50) NOT NULL
);
-- Insert: just omit the id, PostgreSQL fills it in
INSERT INTO roles (role_name) VALUES ('Read Only');
```

### Oracle equivalent (what I'll use here)

```sql
-- 1. Sequence — separate object, generates the next number
CREATE SEQUENCE roles_seq START WITH 1 INCREMENT BY 1 NOCACHE;

-- 2. Table — column is just NUMBER
CREATE TABLE roles (
    role_id   NUMBER PRIMARY KEY,
    role_name VARCHAR2(50) NOT NULL
);

-- 3. Trigger — fires before insert, populates the id from the sequence
CREATE OR REPLACE TRIGGER trg_roles_id
BEFORE INSERT ON roles
FOR EACH ROW
BEGIN
    IF :NEW.role_id IS NULL THEN
        :NEW.role_id := roles_seq.NEXTVAL;
    END IF;
END;
/

-- Insert: same UX as PostgreSQL — omit the id
INSERT INTO roles (role_name) VALUES ('Read Only');
```

### Why three pieces?

Oracle sequences predate PostgreSQL's `SERIAL` and are more flexible 
(they can be shared across tables, cached in memory, incremented manually). 
The trade-off is verbosity. The backlog uses this classic pattern because 
it's what you'll see in most existing Oracle codebases, especially in 
regulated environments.

### Naming conventions for this project

- Sequence: `[table]_seq` — e.g. `roles_seq`, `controls_seq`
- Trigger:  `trg_[table]_id` for auto-increment triggers
- Column:   `[table]_id` — e.g. `role_id`, `control_id`

### Footnote — the modern shortcut (not used here)

Oracle 12c+ supports `GENERATED BY DEFAULT AS IDENTITY`, which compresses 
the three pieces into one like PostgreSQL's `SERIAL`. Deliberately not using 
it — the classic pattern teaches sequences and triggers properly, which I 
need for audit and `updated_at` triggers anyway.

### New syntax this introduces

| Syntax                   | Meaning                                                                      |
|--------------------------|------------------------------------------------------------------------------|
| `:NEW.column_name`       | Inside a trigger, the new row's column value. Colon prefix is Oracle-specific. |
| `:OLD.column_name`       | Inside a trigger, the old row's value (for UPDATE/DELETE triggers).          |
| `/` on its own line      | "End of PL/SQL block, execute it now." Oracle's equivalent of `$$ ... $$`.   |
| `CREATE OR REPLACE`      | Create the object, or replace it if it exists. Idempotent — re-runnable.     |

## 3. Syntax quirks (day-to-day things that bite)

These are the small differences that cause syntax errors every ten minutes 
if you don't know them upfront.

### 3.1 Every SELECT needs a FROM clause

Oracle's SQL parser requires it. Use `DUAL` — a built-in one-row dummy table — 
whenever you don't have a real table to select from.

```sql
-- PostgreSQL
SELECT NOW();
SELECT 'hello' AS greeting;

-- Oracle
SELECT SYSTIMESTAMP FROM dual;
SELECT 'hello' AS greeting FROM dual;
```

### 3.2 String concatenation: `||`

Same as PostgreSQL. Not `+` (that's SQL Server), not `CONCAT()` (that's MySQL-flavoured).

```sql
SELECT 'Hello, ' || first_name || ' ' || last_name AS full_name FROM users;
```

### 3.3 Null coalescing: `NVL` (native) or `COALESCE` (ANSI)

Both work in Oracle. `NVL` is the classic Oracle function and what you'll 
see in existing code. `COALESCE` is ANSI-standard, takes more than two 
arguments, and works across every database — prefer it in new code.

```sql
-- Both return 'unknown' if owner_name is null:
SELECT NVL(owner_name, 'unknown') FROM controls;
SELECT COALESCE(owner_name, 'unknown') FROM controls;
```

### 3.4 No `RETURNING` on plain DML

PostgreSQL lets you do this in one statement:

```sql
INSERT INTO roles (role_name) VALUES ('Admin') RETURNING role_id;
```

Oracle requires PL/SQL and a `RETURNING ... INTO` clause:

```sql
DECLARE
    v_new_id NUMBER;
BEGIN
    INSERT INTO roles (role_name)
    VALUES ('Admin')
    RETURNING role_id INTO v_new_id;
    DBMS_OUTPUT.PUT_LINE('New ID: ' || v_new_id);
END;
/
```

### 3.5 Date / time functions

| Purpose                       | PostgreSQL              | Oracle                      |
|-------------------------------|-------------------------|-----------------------------|
| Current date + time           | `NOW()`                 | `SYSTIMESTAMP`              |
| Current date only             | `CURRENT_DATE`          | `TRUNC(SYSDATE)`            |
| Date arithmetic (add 7 days)  | `NOW() + INTERVAL '7 days'` | `SYSDATE + 7`           |
| Extract year                  | `EXTRACT(YEAR FROM dt)` | `EXTRACT(YEAR FROM dt)` ✓ same |
| Months between two dates      | `AGE()` (returns interval) | `MONTHS_BETWEEN(a, b)`   |

### 3.6 Identifier case sensitivity

Oracle **uppercases** unquoted identifiers. PostgreSQL **lowercases** them. 
Same underlying behaviour (fold to a canonical case) but opposite direction — 
which matters when querying the data dictionary.

```sql
CREATE TABLE roles (role_id NUMBER);
-- Oracle stores this as: ROLES (role_id → ROLE_ID)

SELECT table_name FROM user_tables;
-- Returns: ROLES  (not 'roles')

-- If you ever need case-sensitive identifiers (don't), use double quotes:
CREATE TABLE "roles" (...);  -- now it stays lowercase but you must always quote it
```

Don't use quoted identifiers. Write your SQL in lowercase for readability; 
Oracle handles the folding.

## 4. PL/SQL vs PL/pgSQL

Same ancestor (Ada-derived), same concepts. The differences are mostly 
syntax noise — delimiters, colons, and function names.

### 4.1 Block structure

```sql
-- PostgreSQL (PL/pgSQL) — inside a function body
CREATE FUNCTION my_func() RETURNS trigger AS $$
BEGIN
    NEW.updated_at := NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

```sql
-- Oracle (PL/SQL) — no $$ delimiters
CREATE OR REPLACE TRIGGER my_trigger
BEFORE UPDATE ON my_table
FOR EACH ROW
BEGIN
    :NEW.updated_at := SYSTIMESTAMP;
END;
/
```

### 4.2 `:NEW` and `:OLD` — colon prefix

Inside a trigger, `:NEW` is the row being inserted/updated, `:OLD` is the 
pre-change row. Colons are mandatory in Oracle, absent in PostgreSQL.

```sql
-- BEFORE UPDATE trigger — log what changed
BEGIN
    IF :NEW.status != :OLD.status THEN
        :NEW.updated_at := SYSTIMESTAMP;
    END IF;
END;
/
```

### 4.3 Raising a custom error

| Purpose                | PostgreSQL                         | Oracle                                               |
|------------------------|------------------------------------|------------------------------------------------------|
| Raise a custom error   | `RAISE EXCEPTION 'msg';`           | `RAISE_APPLICATION_ERROR(-20001, 'msg');`            |
| Log a debug message    | `RAISE NOTICE 'msg';`              | `DBMS_OUTPUT.PUT_LINE('msg');`                       |
| User error code range  | n/a                                | -20000 to -20999 (Oracle reserves this range for you)|

```sql
-- Example: block DELETE on the audit_log table (we'll do this in P1-14)
CREATE OR REPLACE TRIGGER trg_audit_log_immutable
BEFORE UPDATE OR DELETE ON audit_log
BEGIN
    RAISE_APPLICATION_ERROR(
        -20001, 
        'audit_log is immutable — UPDATE/DELETE not permitted'
    );
END;
/
```

### 4.4 Anonymous blocks with variables

For ad-hoc scripting — run once, no stored object. Same pattern as 
PL/pgSQL's `DO $$ ... $$` blocks.

```sql
-- PostgreSQL
DO $$
DECLARE
    v_count INT;
BEGIN
    SELECT COUNT(*) INTO v_count FROM roles;
    RAISE NOTICE 'Row count: %', v_count;
END $$;
```

```sql
-- Oracle
DECLARE
    v_count NUMBER;
BEGIN
    SELECT COUNT(*) INTO v_count FROM roles;
    DBMS_OUTPUT.PUT_LINE('Row count: ' || v_count);
END;
/
```

### 4.5 Enabling DBMS_OUTPUT

`DBMS_OUTPUT.PUT_LINE` writes to a buffer that clients read separately. 
In APEX SQL Commands, output appears in a panel below the results — 
usually automatic but if it's missing, run:

```sql
SET SERVEROUTPUT ON;
```

## Quick reference — top 10 gotchas translating PostgreSQL → Oracle

1. `VARCHAR` → `VARCHAR2`
2. `TEXT` → `CLOB`
3. `SERIAL` → `NUMBER` + sequence + trigger
4. `BOOLEAN` → `NUMBER(1) CHECK (col IN (0,1))`
5. Every SELECT needs `FROM dual` if no real table
6. `NOW()` → `SYSTIMESTAMP` (and `CURRENT_DATE` → `TRUNC(SYSDATE)`)
7. No `RETURNING` in plain DML — needs PL/SQL `RETURNING INTO`
8. `NEW.col` → `:NEW.col` (colon prefix, in triggers)
9. `$$ ... $$` → just `BEGIN ... END;` followed by `/` on its own line
10. `RAISE EXCEPTION` → `RAISE_APPLICATION_ERROR(-20001, '...')`

Identifiers fold to **uppercase** in Oracle (vs lowercase in PostgreSQL) — 
write everything in lowercase for readability; Oracle handles the folding.
